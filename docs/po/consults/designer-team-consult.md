## 1. A6 — Schema 동결 검증 게이트

### 1.1 현재 스키마 충분성 평가

**TouchEvent**: `nx, ny, elementID, deviceClass, orientation, timestamp`
**ScrollEvent**: `maxDepth, dwellByBucket, deviceClass, orientation, timestamp` (+ `scrollID`)

| 분석가가 흔히 묻는 질문 | 현재 스키마로 답 가능? |
|---|---|
| "이 화면에서 어디를 많이 누르나" (tap density) | ✅ nx/ny로 가능 |
| "어떤 요소가 클릭당하나" (CTA 식별) | ⚠️ elementID 있지만 정책상 `.hashed` 기본값 → 분석가가 "이 해시가 어떤 버튼인지" 역추적하려면 별도 매핑 테이블 필요. SDK가 주는 건 아니므로 **호스트 앱 책임으로 위임된 상태** |
| "사용자가 스크롤을 얼마나 내리나" (depth) | ✅ maxDepth |
| "어디서 스크롤을 멈추고 오래 머무나" (dwell hotspot) | ✅ dwellByBucket |
| "탭과 스크롤 사이 시간 순서/시퀀스" (행동 흐름) | ❌ 화면 단위 timestamp는 있으나 이벤트 간 순서를 재구성할 키가 없음(세션/시퀀스 ID 부재) |
| "탭이 오발생(mis-tap, rage-tap)인지" | ❌ 연속 탭 간격/재시도 신호 없음 — 단일 탭 좌표만 |
| "스크롤 방향 반전(되돌아가기)" | ⚠️ dwellByBucket은 도달 깊이 기준 누적이라 위로 되돌아간 행동은 구분 안 됨 |
| "한 분석 세션(앱 실행~종료) 내 화면 전환 흐름" | ❌ screenID는 있지만 세션 경계/순서 식별자 없음 |

### 1.2 빠진 필드 후보 (가설 — 검증 필요)

> 아래는 "있으면 좋겠다"는 디자이너 관점 가설입니다. 실제 분석가가 진짜 필요로 하는지는 **A6의 핵심 검증 대상**이지 우리가 단정할 사항이 아닙니다.

1. **sessionID / sequenceIndex** — 탭·스크롤 이벤트의 시간적 순서를 재구성할 키. 없으면 "퍼널 어디서 이탈했나"를 히트맵만으로는 못 봄(별도 raw log 필요).
2. **scroll direction reversal count 또는 backtrack flag** — "헷갈려서 위로 다시 올라갔다"는 UX 신호. 현재 `maxDepth`는 단조 증가 가정처럼 보여서 이 정보가 묻힘.
3. **tap-to-next-screen latency** — 탭 후 화면 전환까지 시간. 느린 반응에 대한 재탭(rage tap) 탐지에 필요할 수 있음.
4. **elementID-to-label 매핑 메타** (SDK가 아니라 호스트 앱 협약 문서) — schema 필드는 아니지만, 동결 전에 "분석가가 hashed elementID를 실제 UI 라벨로 환원할 절차"가 정의돼 있는지는 확인 필요.

🟢 박주니어: "저는 처음엔 'rage tap 플래그도 넣자' 했는데, 이미드님이 '그건 SDK가 판단할 일이 아니라 분석 단계에서 같은 좌표 반복을 보고 추론하면 된다'고 해서 수긍했어요. nx/ny + timestamp만 있어도 분석 쿼리로 rage-tap은 유추 가능하니, 이건 스키마 필드보다 **렌더러/분석 쿼리 레이어의 문제**로 재분류."

🟡 이미드: "sessionID는 좀 다릅니다. timestamp만으로 세션 경계를 재구성하는 건 분석가에게 부담을 떠넘기는 거예요. 이건 진짜 필드 후보로 남겨야 합니다."

### 1.3 동결 전 반드시 검증해야 할 항목

1. **분석가 1~2인 대상 실제 쿼리 워크스루**: "이 스키마로 지난 분기 실제로 받았던 분석 요청 3개를 재현할 수 있는가"를 직접 시연 — 가설이 아니라 실증.
2. **sessionID/sequenceIndex 누락이 실제 블로커인지 확인**: 퍼널/이탈 분석 요구가 있는지 분석가에게 직접 질문. 없으면 동결해도 무방, 있으면 1.0 전에 추가(이후엔 schemaVersion++ 필요해 비용 큼).
3. **elementID `.hashed` 기본값의 역추적 절차 확정**: 분석가가 해시값을 라벨로 되돌릴 방법이 없으면 elementID 필드 자체의 실효 가치가 0에 수렴 — 동결 전에 호스트 앱 쪽 매핑 산출물 존재 여부 확인 필수.

### A6 — Impact / Confidence

| 항목 | 등급 | 근거 |
|---|---|---|
| **Impact** | **Medium** | 스키마 필드 자체는 핵심 인사이트(tap density, scroll depth/dwell)를 이미 커버. 단, sessionID 부재로 "퍼널/이탈 흐름" 분석은 막혀있어 잠재 인사이트 손실 있음. 검증 게이트 통과 시 이 손실을 사전에 막을 수 있다는 점에서 영향력은 "동결 후 후회 비용 회피"에 있음 |
| **Confidence** | **Low** | 실제 분석가 검증 0건. 위 표는 전부 디자이너의 추론이며 실사용 쿼리 패턴 미확인. A6 자체가 "1회 검증을 하자"는 제안이므로 Confidence가 낮은 게 자연스러움 — 검증 후 재평가 필요 |

## 2. A7 — HeatmapViz 렌더러가 enable하는 UX 의사결정

스펙 §4.5 기준(`renderTaps`, `renderScroll`, radius/intensity 파라미터, 화면 배경 오버레이)으로 도출 가능한 의사결정 목록:

1. **CTA/버튼 배치 재배치 판단** — tap density가 의도한 CTA에 집중되는지, 혹은 비활성 영역에 오발생(mis-tap)이 몰리는지 시각 확인 → 레이아웃 변경 우선순위 결정.
2. **스크롤 폼/리스트 화면의 콘텐츠 절단(cut) 판단** — `maxDepth` 히트맵으로 "사용자 N%가 이 지점 이하로 안 내려간다"를 시각화 → 중요 정보를 그 지점 위로 끌어올릴지 결정.
3. **체류 핫스팟 기반 정보 재배열** — `dwellByBucket` 오버레이로 "어디서 오래 머무는가"(이해 어려움 vs 흥미)를 보고 카피/설명 보강 위치 결정.
4. **디바이스/orientation별 레이아웃 회귀 탐지** — `deviceClass` + `orientation` 버킷별 렌더 비교로 "특정 디바이스에서만 깨지는 UX"(예: 작은 화면에서 CTA가 화면 밖으로 밀림) 발견.
5. **A/B 또는 리디자인 전후 비교 판단** — 동일 screenID의 시점별 렌더를 나란히 두고 "레이아웃 변경이 실제 탭 분포를 바꿨는가" 검증.
6. **민감 화면 배제 정책 검증** — 배경 캡처가 제외 화면에서 정상적으로 비어있는지(=프라이버시 정책이 실제 렌더 결과에 반영됐는지) 디자인 QA 관점에서 확인 가능.

🔴 김리드: "이 중에서 진짜 Impact가 큰 건 1번과 2번입니다 — 실제로 화면을 갈아엎는 결정에 직결되니까요. 4번(디바이스별 회귀)은 QA에 가깝지만 디자이너 입장에선 '내가 디자인한 게 실제로 작은 화면에서 안 깨지나'를 보는 유일한 1st-party 수단이라는 점에서 과소평가하면 안 됩니다."

### A7 — Impact / Confidence

| 항목 | 등급 | 근거 |
|---|---|---|
| **Impact** | **High** | 렌더러가 없으면 위 6개 의사결정 전부 "느낌"으로만 판단해야 함. 좌표 정규화(nx/ny) + 디바이스 버킷팅이 이미 스펙에 있어 "정량적 비교가 가능한 시각 자료"라는 가치가 구조적으로 보장됨. 단, "어떤 의사결정에 쓰이는지"는 명확하나 "그 의사결정이 실제로 얼마나 자주/크게 일어날지"는 미검증이라 정량 Impact 수치화는 불가 |
| **Confidence** | **Medium** | 렌더러가 enable하는 의사결정 종류는 스펙(파라미터, API 시그니처)에서 직접 도출되어 추론 근거가 비교적 탄탄함. 다만 "분석가가 실제로 이 렌더 결과를 보고 의사결정을 내리는 빈도/만족도"는 검증 안 됨 → 그 부분만 Low |

## 3. Top-5 분석 화면 (전부 "가설 — 검증 필요")

> 금융앱 일반 UX 패턴 기반 추정. 실제 Finda 앱 분석가 수요 데이터 없음. 우선순위는 사용자 트래픽 추정이 아니라 "히트맵이 줄 수 있는 의사결정 가치가 클 것 같은 화면"을 기준으로 가설화했습니다.

| 순위 | 화면 (가설) | UX 가설 근거 |
|---|---|---|
| 1 | **온보딩/첫 진입 화면** (홈 또는 첫 랜딩) | 첫인상 화면은 이탈률이 전체 퍼널에 가장 큰 영향을 줄 것이라는 일반론. 탭 밀도로 "사용자가 의도한 CTA를 바로 찾는지" 검증 가치가 큼 — **가설, 미검증** |
| 2 | **핵심 전환 폼(대출/상품 신청 등 입력 폼)** | 긴 입력 폼은 스크롤 이탈(dropoff)이 전형적으로 발생하는 지점이라는 금융앱 일반 패턴. `maxDepth`+`dwellByBucket`로 어느 필드에서 이탈하는지 추정 가능 — **가설, 미검증** |
| 3 | **상품 비교/리스트 화면** (대출 상품 리스트 등) | 리스트형 화면은 스크롤 깊이와 탭 클러스터링(어느 카드를 더 누르는지)을 같이 봐야 하는 화면이라 두 인사이트 유형이 모두 적용되는 대표 화면 — **가설, 미검증** |
| 4 | **검색/필터 화면** | 필터 UI는 탭 밀도로 "사용 안 되는 필터 옵션"을 찾아낼 가치가 있다는 가설. 단, 사용 빈도 자체가 낮은 화면일 수 있어 우선순위가 1~3보다는 낮음 — **가설, 미검증** |
| 5 | **마이페이지/설정 화면** | 진입은 잦지만 의사결정 임팩트는 낮을 수 있는 화면. 다만 "사용자가 어떤 메뉴를 찾아 헤매는가"(tap dispersion이 넓으면 정보구조 문제)를 보는 용도로는 가치 있음 — **가설, 미검증**, 우선순위 최하위로 배치 |

🟡 이미드: "5번을 넣을지 고민했어요. 솔직히 설정 화면은 히트맵 안 봐도 IA(정보 구조) 문제인 거 뻔히 보이는 경우가 많거든요. 그래도 '탭 분산도'라는 새로운 신호를 주니까 Top-5엔 남기되 최하위로."

🔴 김리드: "이 리스트 전체에 큰 글씨로 가설 라벨 붙여서 po-team에 전달해주세요. RICE에 실수치처럼 들어가면 안 됩니다."

## 4. 인사이트 유형별 시각 인코딩 권고

### 4.1 탭 밀도 (Tap-density clustering)

- **인코딩**: 가우시안 블러 기반 density heatmap. `HeatmapRenderRequest.radius`(기본 24)로 각 탭 포인트 주변에 커널을 펼치고, `intensity`(기본 0.6)로 누적 시 알파 블렌딩.
- **컬러맵**: **viridis 계열**(어두운 보라 → 청록 → 노랑) 권장. 저밀도=저채도/어두움, 고밀도=고채도/밝음. 빨강-초록 단일축 그라디언트는 채택하지 않음(아래 colorblind 근거).
- **배경**: 기준 스크린샷 위에 반투명(opacity 0.5~0.7) 오버레이 — 민감 화면은 배경 자체가 null이므로 자동으로 빈 캔버스 위 히트맵만 표시(=프라이버시 위반 시각적 신호로도 기능, 빈 배경이면 "이 화면은 배경 캡처가 차단됨"을 분석가가 즉시 인지).
- **상호작용**: hover 시 해당 좌표의 정규화 탭 횟수(raw count는 노출하지 않고 상대 밀도 %만) 툴팁.

### 4.2 스크롤 깊이/이탈 (Scroll-depth/dropoff)

- **인코딩**: 탭 밀도와 시각적으로 명확히 구분되는 형태 — **수평 밴드(horizontal band) 오버레이**. 화면을 depth bucket 단위 가로 띠로 나누고, 각 띠의 채도/밝기로 "그 깊이까지 도달한 사용자 비율"(maxDepth 분포) 표현.
- **dwell(체류)은 별도 레이어**: 띠 위에 점선 텍스처 또는 우측 사이드바 미니 막대그래프로 "이 구간에서 머문 시간"을 추가 표기 — density blur와 같은 시각 언어를 쓰지 않아야 분석가가 "이건 탭이 아니라 스크롤 데이터"임을 즉시 구분.
- **컬러맵**: 탭과 다른 색상 축 사용 — 예: 탭=viridis(보라~노랑), 스크롤=cividis 또는 단일色 blue-scale(진한 남색=낮은 도달, 밝은 하늘색=높은 도달). 같은 화면에 두 레이어를 토글로 전환(동시 노출 X, 인지 부하 방지 — ADHD/판단 피로 원칙 적용).
- **이탈(dropoff) 강조**: 도달률이 급격히 떨어지는 depth bucket 경계에 시각적 마커(가로선 + % 라벨)로 "여기서 N%가 이탈" 표시.

### 4.3 Colorblind-safe 고려

- 적록색맹(deuteranopia/protanopia)이 가장 흔한 유형 — **빨강-초록 대비 축 전면 배제**.
- viridis, cividis, magma 등 perceptually-uniform + colorblind-safe로 검증된 matplotlib 계열 팔레트를 코드 생성(스펙 §8 "리소스 0(컬러맵 코드 생성)")으로 내장.
- 색상에만 의존하지 않기: 탭=blur/원형 클러스터 형태, 스크롤=밴드/선형 형태로 **형태 자체가 다름** → 색각 이상과 무관하게 구분 가능 (디자인 원칙 #5 "색상만으로 상태를 구분하지 않음"과 동일 원칙 적용).

### 4.4 버킷 무결성 (screenID+deviceClass+orientation) UX

🔴 김리드 강조 사항: "데이터 레이어에서 다른 버킷을 안 섞는 건 당연하고, 그걸 **뷰어 화면에 상시 노출**해야 합니다."

- 렌더 화면 상단에 **버킷 식별 배지**(고정 위치, 항상 보임): `screenID · deviceClass(예: 390×844) · orientation(portrait/landscape)`.
- 같은 screenID라도 deviceClass/orientation이 다르면 **자동으로 별도 렌더 + 명시적 화면 전환 UI**(탭으로 전환, 동시 합성 렌더 금지) — 두 디바이스 데이터를 시각적으로 합치면 좌표계가 달라 히트맵이 왜곡되므로 이는 협상 불가 원칙.
- 버킷 전환 시 "다른 버킷입니다"를 색 변화가 아니라 **배지 텍스트 갱신 + 화면 전환 애니메이션**으로 인지시킴(판단 피로 최소화 원칙 — 분석가가 매번 확인하지 않아도 실수로 다른 버킷을 비교하지 않도록 시스템이 막아줌).
